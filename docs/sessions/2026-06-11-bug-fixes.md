# Session: 2026-06-11 — Four bug fixes

## Status: complete

## What was done

### Bug 1 — Subfolder collapse re-expansion

**Root cause A:** `VerticalGroup.tsx` listed `subFolders` in the `useEffect` dependency array for the `subExpandSignal` handler. `subFolders` is passed as `verticals.find(...)?.subFolders` from App.tsx — a new array reference on every parent render. Any state change that re-renders App (star, copy, etc.) produced a new `subFolders` reference, re-firing the effect. If `subExpandSignal` was still set to `{ expand: true }` from a prior Expand All click, this silently re-expanded all subfolders, undoing any manual collapse.

**Fix:** Replaced the `subFolders` dependency with a ref. The ref tracks the latest value without causing the effect to re-subscribe.

**Root cause B:** `toggleGroup` in App.tsx used `prev.has(verticalId)` as the sole expansion check, but `group.expanded` is computed from `prev.has(verticalId) || prev.has(plainId)`. In all-mode, `verticalId` is a composite (`"local::saved-replies"`). If the group was previously expanded in single-source mode, `plainId` (`"saved-replies"`) was still in the set. Clicking to collapse passed the composite key; the old code saw `!prev.has(compositeKey)` and ADDED the composite key, so `expanded` stayed `true`.

**Fix:** `toggleGroup` now derives `plainId` and checks `isExpanded = prev.has(verticalId) || prev.has(plainId)`. On collapse, it deletes BOTH keys.

### Bug 2 — Compact icon grows during drag

**Root cause:** The previous fix (`.compact-view:hover:not(:active)`) was unreliable. `win.setPosition()` calls at 16ms during drag cause Chromium to re-evaluate CSS pseudo-class state. The `:active` flag could be reset by the window repositioning even while the pointer button remained held, reapplying the `:hover` scale.

**Fix:** Added `isDragging` React state. Set on `onPointerDown`, cleared on `onPointerUp` and `onPointerCancel`. The `.compact-view` element gets `data-dragging="true"` while dragging. CSS targets `[data-dragging="true"] .compact-logo` to lock `transform: none` and restore resting opacity for the full drag duration, regardless of CSS pseudo-class state.

### Bug 3 — Pin button on Support Tools items

**Root cause:** The entry-actions redesign session re-added `onTogglePin` and `pinned` props to `SupportToolsGroup` and the pin button to `ToolItem`. Support tool cards are not supposed to be pin-able from the card row (decision from an earlier session).

**Fix:** Removed `pinned` and `onTogglePin` from `SupportToolsGroup` interface and `ToolItem` props. Removed the pin button element from `ToolItem`. Removed both props from the two `SupportToolsGroup` call sites in App.tsx.

Help topic for Support Tools updated: "Favoriting and pinning tools" section simplified to "Favoriting tools" with pin reference removed. "pin" removed from topic tags.

### Bug 4 — Copy count badge on Favorites cards

**Root cause:** `FavoritesView` renders `ResultItem`, which unconditionally renders the `copy-count-badge` span when `entry.copyCount > 0`. Favorites cards should not show the badge.

**Fix:** Added `hideCopyCount?: boolean` prop to `ResultItem`. The badge renders only when `!hideCopyCount && copyCount > 0`. `FavoritesView` passes `hideCopyCount` to all its `ResultItem` instances. The copy tracking logic (`incrementCopyCount`, `most-used` sort) is unaffected.
