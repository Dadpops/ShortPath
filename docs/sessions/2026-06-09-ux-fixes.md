# Session — 2026-06-09

## Goal

Fix UX issues found during testing, add a keyboard shortcuts panel, and enable full keyboard navigation.

## What changed

- **Sample data now visible**: Source filter in rawGroups was excluding `source:"sample"` entries when `activeSource === "local"`. Fixed by including sample entries alongside local ones in that filter.

- **Sync source friendly name in dropdown**: After renaming a sync source in Settings, the label in the header dropdown was stale. Fixed by re-fetching syncSources (and the hotkey) whenever `mode` changes back to `"browse"`.

- **Pin button moved to left side**: The always-on-top pin button moved from `header-actions` (right side) to `app-path-group` (left side), sitting to the left of the "shortpath" path label.

- **Clipboard strip moved to top**: The clipboard strip was at the bottom overlapping the FAB (+) button. Moved it to just below the header (above banners and search). Changed `border-top` to `border-bottom` in the CSS.

- **Notes button now "N"**: The pencil `✎` icon replaced by a monospaced "N" letter button using `header-icon-letter` CSS class.

- **Keyboard shortcuts panel added**: New "K" letter button in header (left of "N") opens a full-screen `KeyboardPanel` component. Shows all shortcuts grouped by category: navigate results, filter/sort, entries, panels, window. Global hotkey is displayed from live settings and styled as `<kbd>` elements.

- **Pin limit removed entirely**: Removed the cap enforcement from `toggle-pin` IPC handler. Removed `pinLimitMsg` state and all its UI from App.tsx. Removed the pin cap toggle from Settings > Behavior.

- **Full keyboard navigation**: Added `Alt+K` (keyboard), `Alt+N` (notes), `Alt+H` (help), `Alt+S` (settings) shortcuts in the window keydown handler. Combined with existing Ctrl+N, Tab (vertical filter), arrow keys, and Esc cascade, the entire app is now navigable without a mouse.

- **Check for updates info**: Added descriptive note below the "Check for updates" button in Settings > Behavior explaining what it does.

- **Version bumped to 0.4.0**.

## Decisions made

- **`source:"sample"` in local filter**: Sample entries belong to the "local" view — they're a placeholder for local data. Excluding them from the local filter was the root bug causing the empty-on-first-launch issue.

- **Sync name refresh via mode effect**: Re-fetching on every mode→browse transition is slightly wasteful but simple and always correct. The alternative (a callback prop chain through SettingsScreen) was more code for no user-visible benefit.

- **Pin limit removal**: The cap was causing confusion and friction. Removed completely rather than raising it — unlimited pins is simpler to explain.
